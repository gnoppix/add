### Part 1: Tuning the Decentralized Hardware Attestation Protocol

To ensure the hardware attestation architecture remains decentralized, peer-to-peer, and resistant to infrastructure backdoors, we must eliminate any dependency on a centralized cloud verification API (such as Microsoft Azure Attestation or online Intel/AMD web endpoints).

Peers must validate each other's hardware state directly using cryptographically bound, deterministic, and localized verification logic.

The refined attestation design addresses this through the following core components:

#### 1. Strict Cryptographic Identity Binding (`REPORT_DATA`)

When a Core Node initializes its headless daemon within a hardware-isolated environment (such as AMD SEV-SNP or Intel TDX), the CPU generates an immutable hardware measurement. To prevent a malicious ISP from intercepting a valid attestation report from a benign machine and replaying it on an unencrypted or compromised virtual machine, the node's long-term network signature key (**ML-DSA-87**) must be inextricably bound to the physical silicon execution layer.

* **The Injection Mechanics:** During the boot sequence inside the Confidential VM, the daemon generates its network-facing `ML-DSA-87` identity public key ($pk_{\text{node}}$). It then hashes this key using SHA-512 and injects the resulting 64-byte digest directly into the CPU's hardware report request via the `REPORT_DATA` field (for SEV-SNP, using the `MSG_REPORT_REQ` guest command).
* **The Result:** The CPU generates a hardware report where the signature covers both the execution state *and* the node's network identity public key. An adversary cannot modify the public key or substitute their own without completely invalidating the hardware signature generated by the CPU's embedded security co-processor.

#### 2. Localized, Decentralized Root Certificate Verification

Instead of pinging a centralized vendor endpoint during a peer handshake, Core Nodes cache the CPU manufacturers' master certificate chains locally.

* **The Trust Anchor:** Core Nodes store the AMD Root Signing Authority (ARK), AMD SEV Signing Authority (ASK), and Intel Product Root Certificate Authority roots natively.
* **Peer-to-Peer Verification:** When Node B requests admission to route traffic through Node A, Node B transmits its hardware report alongside its **Versioned Chip Product Key (VCEK)** certificate. Node A validates the VCEK certificate directly against its local offline copy of the AMD/Intel root keys. Once the VCEK is verified as genuine silicon, Node A verifies the hardware report's signature using that VCEK. This ensures zero runtime calls escape to external tracking servers.

#### 3. Deterministic Software Build Matching (`LAUNCH_MEASUREMENT`)

A verified genuine processor is useless if it is executing a backdoored or modified operating system kernel. The attestation protocol must mandate strict verification of the initial launch hash.

* **The Measurement Vector:** The hardware report exposes a 48-byte or 64-byte vector representing the exact initial state of the memory pages loaded at boot time (`LAUNCH_MEASUREMENT` or `MRENCLAVE`).
* **Open Source Consensus:** The core routing daemon is compiled deterministically (reproducible builds yielding identical binary hashes). The network's certificate-based admission layer maintains an allowed list of these deterministic launch measurement hashes. If an ISP injects an active eavesdropping hook into the kernel, modifies the daemon code, or alters a library block before boot, the `LAUNCH_MEASUREMENT` vector changes, and peers will drop the handshake immediately.

#### 4. Anti-Migration & Migration TCB Invalidation

To prevent an ISP from letting a node boot securely on hardened hardware and then performing a live migration or checkpoint export to an unencrypted, legacy hypervisor, the protocol implements a **TCB Invalidation Loop**.

* **Periodic Re-Attestation:** Admission certificates are given short lifecycles (e.g., 6 hours). Every epoch rollover requires the node to fetch a fresh challenge nonce from its peers, construct a new hardware report containing the nonce in the memory space, and broadcast the verification.
* **Hardware-Reported TCB Constraints:** The report contains the exact patch level and microcode version of the CPU silicon (Current TCB). If the hypervisor forces an unauthorized hardware state modification or attempts an offline state restoration, the mismatch between the expected monotonic counter or microcode profile drops the node out of the active Web of Trust topology.

## PART I: Core P2P Messaging & Metadata Protection

### 1. Advanced Ratcheting: The ML-KEM Braid Protocol (SPQR)
Traditional double ratchets execute instantaneous asymmetric steps because classical Diffie-Hellman public keys are compact (~32 bytes). Post-quantum primitives—specifically **ML-KEM-1024** (~1568 bytes)—introduce significant transport overhead. To prevent conversational channels from stalling due to large key transmissions, the protocol implements the **ML-KEM Braid** paradigm (Sparse Post-Quantum Ratchet).

**The Chunked Key Agreement Flow:**
Instead of blocking transmission to handle a massive public key (`ek`) concurrently, the asymmetric ratchet is pipelined over multiple messages using an incremental interface:
* **The Initialization Spark**: Alice generates an `ek`/`dk` pair. She extracts a **32-byte seed** from the `ek` and computes `Hash(ek)`. She sends only this 64-byte payload to Bob alongside her standard encrypted message.
* **Parallel Generation**: Bob receives the seed and the hash. Using the seed, he can instantly generate the first, massive part of the ciphertext (`ct1`).
* **The Braid Interleave**: As subsequent messages flow back and forth, Alice streams the remaining vector components of the `ek` in small chunks, while Bob streams `ct1`.
* **Reconciliation**: Once both nodes have accumulated the pieces over the sparse network channel, Bob finishes generating the final reconciliation chunk (`ct2`), sends it to Alice, and both sides advance their root key chain.

This allows the protocol to maintain high-strength **Post-Compromise Security (PCS)** without forcing users to experience latency spikes or massive single-packet bursts.

### 2. Metadata Masking: Pure Post-Quantum Sealed Sender
Standard end-to-end encryption masks payload contents but leaves the social graph vulnerable to traffic analysis. The protocol adapts a decentralized "Sealed Sender" layout to ensure intermediate routing or relay nodes cannot map user associations.

**The Cryptographic Envelope:**
When Alice transmits a packet to Bob through the P2P network, she constructs a double-encrypted payload:
1.  **Delivery Tokens**: Bob generates a cryptographically secure 96-bit delivery token derived from a shared secret (exchanged during contact initialization). He registers this token in the local P2P routing space.
2.  **Anonymized Handoff**: Alice hands the packet to the nearest P2P routing nodes *without authenticating her Node ID*. She merely attaches Bob's Delivery Token.
3.  **Relay Processing**: Relays verify that the delivery token is valid for the target area and route it. They have no visibility into who sent the packet, because the sender certificate is entirely sealed inside the envelope encrypted with Bob's public key.

### 3. Scalable Group Messaging: PQ-Sender Keys
Encrypting a single message individually for every recipient in a large group chat (pairwise encryption) breaks down in a peer-to-peer network due to bandwidth constraints. The system adapts a Sender Keys protocol, hardened for a post-quantum framework.
* **Group Genesis**: When a group is formed, every member generates a local **Symmetric Chain Key** and a signing keypair (**ML-DSA-87**).
* **Key Distribution**: Using the pairwise ML-KEM Braid channels already established between members, each user securely broadcasts their Sender Key bundle to the other group participants.
* **Message Fan-out**: When a member sends a message to the group, they encrypt the payload *exactly once* using an AES-256-GCM key derived from their own Sender Key chain. They then sign the ciphertext with their ML-DSA key and broadcast it to the P2P network swarm.
* **Ratchet Advancement**: Every time a member sends a group message, they locally advance their symmetric KDF chain. If a member leaves or is removed from the group, a collective epoch reset occurs: all remaining members generate brand new Sender Key bundles and distribute them to enforce forward and backward isolation.

### 4. Privacy-Preserving Contact Discovery
Because a P2P network cannot rely on centralized hardware, contact discovery is managed via **Private Information Retrieval (PIR)** over the distributed network.
* **Blind Registries**: Users hash their discovery vectors (e.g., an email or unique handle) using a salted, iterated cryptographic hash function, storing the resulting locator token in the DHT.
* **Zero-Knowledge Queries**: To check if a contact is online, a node initiates an asymmetric PIR query against the network segment storing that locator token. The network node handling that sector returns the corresponding PQC identity public key without ever learning which specific identity the initiator was searching for, keeping the user's address book completely local and undisclosed.

## PART II: Mobile, Bandwidth & Push Architecture

High-security metadata-hiding techniques (like continuous cover traffic and mixnets) are inherently hostile to battery life and data caps. To adapt the protocol for mobile environments without completely degrading its cryptographic integrity, the specification transitions from a flat, homogenous P2P model to an **Asymmetric Edge-Core Architecture** with adaptive, budget-aware signaling.

### 1. Asymmetric Network Topology: Mobile Edge vs. Core Nodes
Mobile devices cannot act as full P2P routing infrastructure. Forcing a smartphone to route third-party Sphinx packets or participate continuously in Kademlia DHT maintenance will exhaust its battery and data pool within hours.
* **Core Nodes (Infrastructure Tier):** Stationary devices (desktops, home servers, dedicated nodes) with unmetered connections. They handle the mixnet routing, host the Oblivious Hash Tables (OHT), and buffer message drops.
* **Edge Clients (Transient Tier):** Mobile or data-constrained devices. They operate purely as *leaf nodes*. They never route traffic for others, never store DHT keys, and interact with the network via an asynchronous poll/push mechanism.

### 2. Bandwidth-Aware Signaling & Dynamic Cover Traffic
Constant-rate signaling (sending dummy packets 24/7) is unviable on limited data plans. The protocol implements an **Adaptive Traffic Budgeting Engine** that queries the OS network state.

| Network State | Cover Traffic Strategy | Bandwidth Footprint | Security Trade-off |
| --- | --- | --- | --- |
| **Unrestricted (Wi-Fi / Charging)** | **Continuous Poisson Stream:** Constant background dummy generation. Full mixnet integration. | High (~50–100 MB/hr) | Maximum protection against Global Passive Adversaries (GPA). |
| **Metered / Cellular (Normal)** | **Burst-Padding Only:** Zero background traffic when idle. When a real message is sent, it is padded with dummy blocks to a uniform size. | Low (Proportional to usage) | Vulnerable to idle timing analysis, but protects message length and internal structure. |
| **Tactical / Critical Low Data** | **Deferred Queueing:** All non-essential keys, sparse ML-KEM Braid updates, and profile synchronization are compressed and cached locally. Packets are flushed only in batches. | Minimal | Protects content securely; metadata protection is minimized to favor survivability. |

### 3. Privacy-Preserving Post-Quantum Push Notifications (PQ-PPN)
Mobile operating systems aggressively terminate background processes to save power, breaking long-lived P2P TCP/UDP sockets. To receive messages while backgrounded, mobile clients use push notification networks (APNs/FCM) or decentralized alternatives (UnifiedPush/ntfy) via an untrusted **Push Proxy**.
* **Blinded Push Tokens**: The mobile client selects a Core Node to act as its Push Proxy. The client generates a single-use cryptographic token using a zero-knowledge scheme and registers it with the proxy.
* **Zero-Knowledge Triggers**: When a message arrives at the Oblivious Hash Table for the mobile node, the storage node notifies the Push Proxy. The proxy sends an entirely opaque push payload through Google/Apple infrastructure. The OS wakes up the messaging daemon in the background, allowing it to open a brief, encrypted connection to the P2P storage node, download the payload, and close the socket immediately.
* **Privacy Preservation**: Google, Apple, or intermediate cellular carriers see only that a push event occurred. They cannot read the message contents, identify the sender, or link the push token to a persistent cryptographic identity.

### 4. State-Compressed ML-KEM Braiding
The **ML-KEM Braid Protocol** must be optimized to prevent transport-layer overhead from consuming metered data.
* **Pre-computed Seed Caching**: Instead of generating and negotiating new key material over cellular paths, mobile nodes pre-compute batches of public key seeds and exchange them *only* when connected to unrestricted Wi-Fi.
* **Ratchet Slow-Down**: While on cellular networks, the **Asymmetric Ratchet** interval is extended. Instead of advancing the ML-KEM root key on every single message turnaround, the session relies longer on the low-overhead **Symmetric Ratchet** (AES-256-GCM key rotation), costing zero extra bytes per message. The heavier asymmetric post-quantum step is deferred until a specified threshold of messages is met or an unrestricted network connection becomes available.

## PART III: Local Data-at-Rest Protection (Mobile PQ Vault)

Securing a post-quantum P2P client on a mobile device requires defending against physical access threats (device theft, flash memory extraction, or forensic imaging) without overtaxing mobile Application Processors (AP). Because embedded Hardware Security Modules (HSMs) are throughput-constrained, a **hybrid hardware-software cryptographic key layout** is required.

### 1. Hardware-Bound Cryptographic Key Hierarchy
The local client isolates highly sensitive post-quantum identity keys (**ML-DSA-87**) and active session states behind a layered key-encryption-key (KEK) architecture.
* **Hardware Extraction Barrier**: The client requests the device's native HSM to generate a classical 256-bit symmetric key utilizing hardware-level isolation. This key never leaves the secure processor's isolated RAM.
* **User Entropy Guard**: When the user opens the application, they provide local entropy (a high-entropy passcode or a biometric signature verified directly inside the secure processor). This inputs into an **Argon2id** key derivation function to generate $K_{\text{User}}$.
* **The Master Storage Key**: The final operational key used to unlock local state storage is derived by combining both secrets via HKDF-SHA-512. This ensures that even if an adversary desolders the flash memory chips or extracts a full raw image of the storage layer, the data remains cryptographically dead. The physical, device-specific silicon UID is required to construct the key loop.

### 2. On-Disk Database Architecture: Hardened SQLCipher
All persistent records are structured inside an embedded SQLite database encrypted via SQLCipher.
* **AEAD Enforcement**: The storage engine discards standard CBC mode in favor of **AES-256-GCM**. This provides authenticated encryption with associated data (AEAD), blocking localized ciphertext tampering or bit-flipping forensic attacks.
* **Page-Level Randomization**: The database is compiled with a 4096-byte page size. Each individual sector page is encrypted using a unique 96-bit nonce derived from an isolated page-specific salt vector, preventing patterns from being indexed across structural data tables.
* **In-Memory Quarantine Tiers**: To safeguard uncommitted cryptographic handshakes over flaky cellular networks, the client instantiates an unbacked ephemeral database (`:memory:`). Partial ML-KEM private state materials are stored exclusively in temporary RAM and never write directly to physical flash cells until key execution is fully finalized.

### 3. Ephemeral Memory Control & Biometric Gates
Because mobile operating systems frequently background apps and freeze process states, memory preservation is critical to prevent leaks via physical memory dump attacks.
* **Active Memory Shredding**: The instant the application transitions to the background or the OS signals a device lock event, the application daemon initiates a memory flush. The master storage key and open SQLCipher handles are aggressively scrubbed from application RAM using compiler-optimized memory-zeroing functions (`memset_s`).
* **Biometric Access Lifecycles**: Access to database decryption handles is gated behind explicit hardware flags. If the app process attempts to query historical data after an idle timeout without a fresh biometric validation handshake, the security framework throws a hard runtime fault and instantly terminates the local memory loop.

### 4. Anti-Forensic Rollback & Cloning Protections
A key risk with mobile data acquisition tools is the cloning of a device's file system state to replay cryptographic handshakes or brute-force state machines offline.
* **Lattice Key Blinding**: When long-term ML-DSA identity public/private key pairs are pulled from disk to sign P2P payloads, they are held in memory using **additive secret sharing (masking)**. The private key polynomial vectors are split into randomized components across the heap space to prevent microarchitectural side-channel sniffing or debugger exploitation.
* **Hardware Monotonic Counter Binding**: The state sequence index of the local KEM-Based Double Ratchet is cryptographically paired with the host device's internal hardware-backed monotonic counter. If an investigator clones the file system configuration and attempts to run it on an emulated instance or a separate test device, the mismatch between the database state index and the physical secure processor's hardware counter triggers an automated state-destruct sequence, completely purging the header pages of the local database.

## PART V: Real-World Implementation Defenses & Optimization Engine

### 1. Coordinated Baseline Noise Protocol (CBNP) for Mixnet Bootstrap
To solve the **"Cold Start" Problem** where low early-stage traffic volume exposes early adopters to statistical correlation attacks, Core Nodes implement a unified background padding framework.
* **Coordinated Dummy Loops**: When a Core Node is certified and joins the backbone, it does not wait for user traffic. It initiates a continuous, low-overhead baseline loop of synthetic Sphinx packets sent to adjacent Core Nodes.
* **Decentralized Volume Anchoring**: The volume of background noise is calculated dynamically based on active certificate counts. If user traffic drops below a mathematically proven anonymity threshold ($A_{\text{min}}$), the Core Nodes automatically scale up synthetic padding to maintain a flat, uniform network-wide packet velocity. 
* **Adversary Blindness**: To an outside observer or GPA, the network exhibits an identical, stable cryptographic throughput profile from Day One, shielding early users before the software achieves mass deployment.

### 2. Speculative OMAP Pipelining & Bloom-Filtered Delta Syncing
To mitigate high **Edge Latency** inherent in tree-based Oblivious RAM (ORAM) or Oblivious Maps (OMAPs) over unstable 4G/5G mobile links, the edge client bypasses sequential tree-traversals.
* **Parallel Asymmetric Lookups**: Instead of performing successive $O(\log N)$ round-trips to address lookup collisions or query data blocks over a high-latency WAN, the client pools index requests. It generates a single batch pipelined query containing real and targeted dummy access indexes, sending them in a single transport layer blast to distinct nodes in the core storage cluster.
* **Bloom-Filtered Delta Syncing**: Mobile edge nodes store a local, compact **Bloom Filter** representing their cached mailbox state. When polling an OHT storage node, they do not read the database structure; they transmit the Bloom Filter hash matrix. The storage node performs an oblivious delta comparison and streams back only the missing message blocks, compressing interaction rounds down to an effective $O(1)$ workflow.

### 3. Hardened Secure Subspaces & Compile-Time Assembly Guarantees
To prevent cross-platform memory leaks, OS swap vulnerabilities, or aggressive compiler optimization passes that compromise local data zeroing, the native C++/Rust layer bypasses standard allocation abstractions.
* **OS Paging Protection (`mlock` / `VirtualLock`)**: Cryptographic keys, temporary private lattice variables, and SQLCipher handle segments are allocated exclusively inside isolated memory pools bound via `mlock` (Linux/Android) or `VirtualLock` (Windows) / `mlock` constraints (iOS). This forbids the host operating system from writing sensitive RAM buffers out to unencrypted physical storage or swap partitions during background suspended cycles.
* **Virtual Guard Pages**: Key memory pools are bound by virtual memory allocation maps (`mmap`/`VirtualAlloc`) wrapped on both boundaries with unmapped guard pages (`PROT_NONE`). Any accidental pointer arithmetic error or buffer overflow attempt results in an immediate hardware page-fault and instantaneous process self-termination before memory extraction can occur.
* **Defeating DSE with Assembly Barriers**: To ensure compiler optimization passes (such as LLVM's Dead Store Elimination) never strip away the secure zeroing code during production builds, the memory-scrubbing loop embeds a physical inline assembly memory barrier (`asm volatile("" ::: "memory")` or explicit volatile pointer assignment). This forces the compiler toolchain to treat the zeroing sequence as a volatile side-effect that cannot be bypassed.

/// Securely overwrites a memory buffer with zeros.
/// Bypasses Dead Store Elimination (DSE) optimizations using volatile writes and memory barriers.
#[inline(never)] // Protect stack layout isolation
pub fn secure_zero_memory(buffer: &mut [u8]) {
    let mut ptr = buffer.as_mut_ptr();
    let end = unsafe { ptr.add(buffer.len()) };

    while ptr < end {
        unsafe {
            // Volatile write ensures the compiler cannot optimize this away
            std::ptr::write_volatile(ptr, 0u8);
            ptr = ptr.add(1);
        }
    }
    
    // Explicit sequential-consistency fence acts as an architectural memory barrier
    std::sync::atomic::compiler_fence(std::sync::atomic::Ordering::SeqCst);
}

        secure_zero_memory(&mut self.key_material);
    }
}
